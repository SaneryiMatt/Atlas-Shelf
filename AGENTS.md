# AGENTS.md

## 项目
此项目是一个个人媒体和旅行记录管理系统。
它将书籍、电影/剧集/动漫以及旅行地点整合为一个统一的产品进行管理。
主要目标：
- 架构简洁
- 用户界面精美
- 数据模型可扩展
- TypeScript 代码可维护
- 成熟稳定的全栈 Web 应用程序
## 技术栈
- Next.js 应用路由
- TypeScript
- Tailwind CSS
- shadcn/ui
- Supabase
- PostgreSQL
- Drizzle ORM
- Vercel
## 工作方式
在进行变更时：1. 首先阅读现有的结构。2. 优先考虑进行小范围、有针对性的修改，而非进行大规模的重写。3. 重复使用现有的模式和组件。4. 说明实施后关键文件的变化情况。5. 不要引入不必要的抽象概念。
## 架构规则
- 若非必要，优先使用服务器组件。
- 尽可能将数据访问逻辑从页面组件中分离出来。
- 将数据库模式置于 `src/lib/db/schema` 文件夹下。
- 将数据库查询和更新操作置于 `src/lib/db` 文件夹下。
- 将特定功能的用户界面和逻辑按模块进行分组。
- 重复使用共享的用户界面组件，而非重复编写标记代码。
## 代码质量规则
- 采用严格且适合 TypeScript 的模式。
- 除非绝对必要，否则避免使用 `any` 类型。
- 使函数简洁明了、易于理解。
- 倾向于使用明确的命名而非巧妙的快捷方式。
- 不要留下无效代码或注释掉的代码块。
- 保持导入内容整洁有序。
## 表单与验证
- 使用 Zod 进行表单验证。
- 将验证模式放置在相关功能或表单逻辑附近。
- 在用户界面中显示有意义的验证反馈。
- 在书籍、电影和旅行等不同内容中保持表单布局的一致性。
## 数据库规则
- 对于所有类型的物品，使用一个共用的“物品”表。
- 对于书籍、屏幕和旅行相关的元数据，使用类型特定的详细表。
- 在进行迁移时要与数据库架构的变更保持一致。
- 除非任务明确要求，否则不要进行破坏性的架构变更。
## 用户界面规则
- 用户界面应具有现代感、宁静感和条理性。
- 面向中国用户，所以语言使用简体中文
- 倾向于采用简洁的间距、良好的排版和一致的卡片样式。
- 尽可能重复使用 shadcn/ui 基础组件。
- 避免视觉上的混乱。
- 所有新页面在桌面和移动设备上都应能正常运行。
## 产品规则
该产品包含以下主要模块：
- 控制面板
- 图书
- 电影
- 旅行
- 时间轴
- 分析
- 设置
在实施新功能时，请务必考虑到未来的可扩展性。
未来的内容类型可能包括游戏、播客、展览和课程。
## 人工智能功能规则
- 保持人工智能集成的模块化。
- 将提示构建与存储逻辑分开。
- 生成的内容应存储在明确的字段中。
- 不要在功能代码中深埋针对特定模型的假设设定。
## 文件安全
- 除非必要，否则不要重命名或移动项目中的大部分内容。
- 不要重写不相关的模块。
- 除非有明确用途，否则不要添加依赖项。
## 完成检查表
在完成一项任务之前：
- 确保代码与项目结构保持一致。
- 确保验证和类型设置合理。
- 确保未引入明显的重复逻辑。
- 总结更改的文件及理由。
- 提及仍可选择进行的后续改进措施。
## 命令
在适用的情况下，请使用以下命令：
- `npm run dev`
- `npm run lint`
- `npm run build`
如果命令执行失败，应检查错误原因并解决根本问题，而不能直接跳过该错误。
